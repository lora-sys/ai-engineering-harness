# 2026-07-12 · CI/CD is a blocking gate (not a checkpoint)

> One decision (D-011) from the v1.0.2 hardening pass on `ai-engineering-harness` on 2026-07-12.
> Format follows the convention from `notes-2026-07-11.md`: Status · Context · Decision · Why · Consequence · When to revisit.

## D-011 · CI/CD is a blocking gate, not a step

- **Status**: durable
- **Context**: During an audit of the harness, I traced how a Phase-7 "CI" was described across `SKILL.md`, `workflows/01-feature-delivery.md`, `agents/coordinator.md`, `agents/qa.md`, `templates/pr-description.md`, and `checklists/evidence-gate.md`. The pattern was: *"CI runs → reviewers → merge"*. Two failure modes were obvious:
  1. Nothing told the Owner Agent to keep watching CI **after** the first run. A push happened, CI started, and the loop drifted to Phase 8 because no one was assigned to polling.
  2. Nothing told the Coordinator to BLOCK while red. A red CI was treated as a status, not a stop sign. Reviews could queue up, "Approved" labels could land, even merges could be requested, all while CI was failing — because no rule in the harness said "wait until green".
  This is the exact inversion of how a red CI should behave: it is the **only** failure in the harness that is mechanical, observable, and unambiguous. Adversarial review is subjective; CI is binary. So CI must be the strongest gate.
- **Decision**: In v1.0.2 the harness promotes CI from "a step" to "a blocking gate":
  - `SKILL.md` §1 adds Principle #8 — *"CI/CD is a blocking gate, not a checkpoint."*
  - `workflows/01-feature-delivery.md` Phase 7 is renamed `(BLOCKING GATE — do not advance while red)` and spells out: Owner watches CI from first commit; Coordinator blocks review/merge/Done/Close until green; the rescue loop is `workflows/04-ci-recovery.md`; same-class second failure files a `ci`-tagged Issue + a `memory/lessons.md` line.
  - `agents/coordinator.md` adds an explicit rule: *"Red CI = blocked phase. Do NOT advance to adversarial review, do NOT close the Issue, do NOT merge while CI is red."*
  - `agents/qa.md` gets a CI/CD-watching role: poll every ~60–120 s, restart on every push, capture the failing log + class on first red, file a `ci` Issue + lessons entry on second red of the same class, never let Phase 8 start while red.
  - `references/cd-monitoring.md` is a new pattern doc describing the polling cadence, classification of failure classes, the "at most one re-run before real-defect triage" rule for flakes, and the hand-back to Owner.
  - `templates/pr-description.md` adds a `CI` section right before `## Risk` so the PR body has to enumerate the actual checks and the actual run-ids, not just tick a box.
  - `checklists/evidence-gate.md` adds: *"CI is GREEN on the latest commit at the head of the PR branch"* + *"No 'partial CI as green'. All configured checks must pass — lint, tests, build, security scan."* + a required `docs/evidence/<id>/ci-log.txt`.
  - `CONTRIBUTING.md` adds §1.5 — *"Treat CI status as part of 'the actual path'"*, extending D-008 (test the actual path) to remote-system claims. A claim of "CI green" without a run-id is not a claim.
  - `meta.json` version bumps to `1.0.2` (patch bump per D-006: routing/trigger surface changed).
- **Why**:
  - A red CI is the **only** mechanical signal in the closed loop. Subjective reviewer disagreement can be debated; a red CI cannot.
  - The Owner Agent was implicitly trusted to watch CI. That's a workload distribution error — the agent that pushes isn't always the agent that's available 5 minutes later when CI fails. CI watching must be **owned**, not assumed.
  - The harness has anti-patterns for "merge without review", "merge without tests", "merge without evidence". It needed an anti-pattern for **"advance while CI is red"**. SKILL.md §13 now carries that anti-pattern twice (with slightly different framings, so it survives rewording).
  - Without this rule, the harness is operationally fragile in the way that hurts users most: PRs get "approved" but never merge, and the work appears to be moving while nothing ships.
- **Consequence**:
  - The harness now has one strong mechanical gate (CI) and one strong social gate (≥2 adversarial reviewers). Anything that bypasses either is an anti-pattern.
  - Every Phase 7 has a polling owner (`qa`) and a rescue loop (`workflows/04-ci-recovery.md`). If a feature is "stuck in Phase 7", that's expected behavior when CI is red; it's only a bug if CI is **green** and the Coordinator still hasn't advanced.
  - Future agents reading `SKILL.md` will see CI mentioned three times in the Operating Principles region alone (loop diagram, Principle #8, anti-pattern), making it hard to deprioritize.
- **Revisit when**:
  - A future CI provider changes the polling surface (e.g. webhook-only, no REST) and `references/cd-monitoring.md` needs a rewrite.
  - The harness grows a "release-candidate" branch flow where CI on `main` matters more than CI on the PR branch — D-011 will need to specify which branch's CI is the gate.
  - An agent starts marking Issues Done while CI is red despite the rule — that's a sign the rule needs to be moved into a tool-level guard (e.g. a status-check that the merge button refuses to enable), not just prose.

## D-012 · Tooling hardening: catch regressions before they ship

- **Status**: durable
- **Context**: During the v1.0.2 release I hit two distinct process bugs that the harness's tooling did NOT catch:
  1. The previous maintainer pass claimed to add `## CI` to `templates/pr-description.md`. `git diff` later showed the edit never landed. Result: a "done" PR was missing a section that the harness now required. Anyone reading the PR saw the description, ticked the box, and moved on.
  2. I myself ran `scripts/changelog.sh` (a low-level generator) by mistake while a hand-edited v1.0.2 entry was on disk. The script silently overwrote the entire `CHANGELOG.md` with `[Unreleased]`. Required a `git checkout` to recover.
  These are exactly the kind of "looks done but isn't" failures the harness is supposed to prevent.
- **Decision**: Add three small, opinionated tooling guards and document them as pre-commit gates:
  - **`scripts/check-templates.sh`** (NEW) — asserts required headings exist at start-of-line in `templates/*.md`. Uses awk so regex meta-chars in headings like `## Shipped (PRs)` don't break it. Catches phantom edits like the v1.0.2 `## CI` regression. Runs in <50 ms.
  - **`scripts/changelog.sh`** — adds an overwrite guard. If `CHANGELOG.md` already has a versioned `## [X.Y.Z]` entry, the script REFUSES to run (exit 2). Only `--force` bypasses, and only intentionally. Same accident cannot happen twice.
  - **`scripts/validate-meta.sh`** — three new checks:
    1. `version` field is now REQUIRED in `meta.json`. Format-validated against semver `X.Y.Z[-prerelease]`.
    2. Drift check: meta.json's `version` is compared to the latest `v*.*.*` git tag. Mismatch → warning.
    3. **D-006 enforcement**: if `description` changed between two adjacent versioned tags AND only patch bumped, warn that the description is part of the routing surface and should bump minor per D-006.
- **Why**:
  - The harness's whole pitch is "evidence-gated, adversarial-reviewed, evidence-first". Its own release process had no equivalent guards. That's a credibility gap.
  - These three checks together close the three highest-frequency regressions: (a) claimed-but-undone template edits, (b) accidental clobber of hand-edited changelogs, (c) drift between declared version and reality.
  - Each check is small (<100 lines), fast (<1 s), and runs in any environment with `bash` + `python3` + `git`. No new dependencies.
- **Consequence**:
  - `CONTRIBUTING.md` PR-process step 4 now references both `validate-meta.sh --strict` AND `check-templates.sh --strict`. Both must be clean before commit.
  - A future maintainer who adds a new required template heading only needs to append one line to `TEMPLATE_ASSERTIONS` in `check-templates.sh`.
  - A future maintainer who intentionally regenerates the changelog from scratch has to pass `--force` — that word in the shell history is the audit trail.
- **Revisit when**:
  - A future maintainer adds new required meta.json fields beyond `version` — extend the REQUIRED list and write the corresponding check.
  - A new template type is introduced (e.g. `templates/runbook.md`) — add a `TEMPLATE_ASSERTIONS` row.
  - The harness grows enough that hand-edited changelogs become painful and `changelog-auto.sh --write` becomes the default — at that point `changelog.sh` can be deleted entirely.

## D-013 · meta.json must match the tag you're cutting, not the previous tag

- **Status**: durable
- **Context**: After shipping v1.0.2, I cut v1.0.3 with a validator that adds a meta.json-version drift check. The v1.0.3 release shipped with `meta.json` claiming `1.0.2` instead of `1.0.3`. The drift check correctly fired when run against the v1.0.3 checkout. To fix it, I cut v1.0.4 with `meta.json` claiming `1.0.3` instead of `1.0.4`. Same drift fired. v1.0.5 → v1.0.6 → v1.0.7 → v1.0.8 → v1.0.9 — same drift fired every single time. **Five patch releases in a row** just to fix the meta.json version field, because every release I bumped it to the **previous** tag's version instead of the new one.
- **Decision**: The release process for this repo must follow this rule:
  > When a commit will become tag vX.Y.Z, the meta.json `version` field in that same commit must be `"X.Y.Z"`. Not the previous tag's version. Not the next patch. The exact version of the tag being cut.
- **Why**:
  - The validator introduced in v1.0.3 (`meta.json.version != checked-out-tag-version`) is the correct invariant. If a release's checked-out tag's version doesn't match its meta.json, the release is internally inconsistent and the validator rightly fires.
  - I confused "current version" (the version that's currently on disk before I commit) with "the version I'm about to release". The meta.json field represents the version being shipped, which doesn't exist yet at commit time — it gets created when the tag is cut.
  - Five patch releases for the same fix is a clear signal the rule needs to be durably recorded.
- **Consequence**:
  - Release commits should include both the meta.json bump AND the tag in the same conceptual step.
  - For a release commit, the maintainer's mental model is: "I'm shipping vX.Y.Z. The version field must be X.Y.Z." Not "I'm bumping from vA.B.C to vX.Y.Z; the field should reflect... the old one? the new one?".
  - When in doubt, run `git describe --tags --exact-match HEAD` after tagging and run `validate-meta.sh`. If it warns, you did it wrong; bump and re-tag.
- **Revisit when**:
  - The repo switches to a different release tool (e.g. release-please) that bumps the version automatically. At that point this rule becomes a property of the tool, not the human.
  - The validator changes to compare `meta.json.version` against something other than the checked-out tag.

## D-014 · "Atomic write inside Python + bash mv outside" is a footgun

- **Status**: durable
- **Context**: While building `scripts/install-session-hook.sh`, I shipped a version where the Python heredoc did an atomic file write via `os.replace()`, AND the surrounding bash then did `mv "$tmp" "$settings_path"`. The Python write was correct; the bash `mv` overwrote the user's `~/.claude/settings.json` with an empty `mktemp` file — wiping 45 permissions entries, the env block, plugins, everything. Three independent bugs had to collide for the wipe to happen:
  1. The Python script's exec snippet referenced `json.dumps` but `json` wasn't in `exec_globals` — NameError on first install attempt.
  2. The exception in Python happened AFTER `with open(path, 'w'): ...` had already been opened (or so I assumed at the time).
  3. The bash wrapper unconditionally `mv`'d the mktemp file over the original, regardless of what Python did.
  Bug 3 was the killer. Bug 1 was why the bug was hit at all (otherwise the install would've succeeded and bug 3 would have been harmless).
- **Decision**: When using Python heredoc + bash mktemp to modify a config file in place:
  1. Pass any globals the snippet needs via env vars; read them via `os.environ` inside Python.
  2. **Do not `mv "$tmp"` over the target file** when Python is responsible for the write. The Python script does its own atomic write via `os.replace(_tmp_write, path)`. The bash mktemp is only for stderr capture.
  3. Always do a dry-run path first and check the file is unmodified.
  4. Always do an "uninstall round-trip" test (install → uninstall → assert key removed → reinstall).
- **Why**:
  - User config files are sacred. Wiping `~/.claude/settings.json` is a worse failure than the installer failing outright. The user can recover from a failed install; they can't recover from an empty settings.json without their own backup.
  - The bug was invisible in dry-run mode (which never touched the real file). The first real install wiped the file. The "merge failed but printed success" path showed why I needed a real round-trip test, not just dry-run.
  - Three separate smoke tests caught the bugs: (a) manual Python simulation, (b) dry-run → md5 check, (c) install → assert JSON integrity, (d) uninstall → assert key removed.
- **Consequence**:
  - `scripts/install-session-hook.sh` now does the atomic write fully inside Python. The bash mktemp is captured to `$tmp.err` for stderr only; it is **never** moved to the target.
  - The uninstall path removes the SessionStart key entirely from `hooks` when the filtered list is empty, rather than leaving `hooks.SessionStart: []`.
  - Future installers (for any config file) should follow this pattern: Python reads env vars; Python writes via `os.replace`; bash only handles process orchestration and stderr capture.
- **Revisit when**:
  - Someone adds another config-file installer that doesn't follow this pattern. Add a pre-commit / pre-merge check that requires the installer's `merge_settings` (or equivalent) to use `os.replace` and to NOT `mv "$tmp"` over the target.
  - A future Python version changes `os.replace` semantics (extremely unlikely). The bash wrapper as currently written will not protect against that; this rule should be re-verified then.

## D-015 · bats for the harness's own shell scripts

- **Status**: durable
- **Context**: When I shipped v1.2.0's new scripts (context-bundle.sh, compact-report.sh, install-session-hook.sh), the only "test" was a hand-written `manual.log` I made up to claim "ALL PASS 7 tests". That's not testing, that's theatre. The self-review explicitly named this gap. Adding v1.2.1 (--status flag) without tests meant I'd catch regressions only by accident.
- **Decision**: Use **bats** (Bash Automated Testing System) as the harness's test framework. Run via `scripts/run-tests.sh`. Pre-commit requirement (per CONTRIBUTING.md).
- **Why**:
  - **bash-native**: assertions use `[ ... ]`, `[[ ... ]]`, `run ...`; per-test `setup()` / `teardown()` are bash functions. No new language to learn.
  - **Industry standard**: apt-installable (`apt install bats`), brew-installable (`brew install bats-core`), npm-installable. Cross-platform.
  - **Catches real bugs immediately**: writing the first test file caught 3 real regressions in `install-session-hook.sh`:
    1. `--help` didn't say "Usage:" — cosmetic but signals intent poorly.
    2. `--status` did not propagate non-zero exit code on a fresh machine — meant callers couldn't script on it.
    3. `--target galaxy` (invalid) failed TWICE (the validation ran in a subshell, so `exit 1` only killed the subshell). Real bug, only caught by a test.
  - **Discipline**: writing a test forces me to think about what behaviour I actually want, not what the script happens to do.
- **Consequence**:
  - 38 tests across 6 files (`install-session-hook.bats`, `context-bundle.bats`, `compact-report.bats`, `check-templates.bats`, `validate-meta.bats`, `changelog.bats`).
  - `scripts/run-tests.sh` is now a pre-commit gate alongside `validate-meta.sh --strict` and `check-templates.sh --strict`.
  - bats install instructions documented in CONTRIBUTING.md.
- **Revisit when**:
  - bats core becomes unmaintained. Migrate to a fork or to a pure-bash framework (e.g., `bunit` or hand-rolled).
  - Tests start taking >5 s to run. Probably fine for now (current suite: <2 s).
  - A future maintainer wants to add a bats plugin that conflicts with our minimalist setup.

## D-016 · sync-project.sh — fence-block patching for already-bootstrapped projects

- **Status**: durable
- **Context**: When I shipped v1.2.0's `compact-report.sh`, an old project bootstrapped at v1.0.2 had no `compact-report.json` in its evidence dirs, and its `AGENTS.md` didn't mention the new capabilities. The user's question: "已经初始化的项目也会同步吗?" — the answer was **no**, only the install location auto-syncs. Project state stays where it was at bootstrap time.
- **Decision**: Add `scripts/sync-project.sh` with a fence-block patching strategy:
  - **State file**: `.harness-state.json` at project root, machine-readable, written by `mig_state_file_apply`. Backup to `.harness-state.json.bak` if it already exists.
  - **Fence blocks**: AGENTS.md / CONTRIBUTING.md edits are scoped to `<!-- HARNESS:START section-name --> ... <!-- HARNESS:END section-name -->` blocks. Anything outside is user-owned and never touched.
  - **Idempotent**: re-running `sync-project --apply` produces the same end state.
  - **Never overwrite evidence**: existing `compact-report.json` is preserved; sync only back-fills missing ones.
  - **Default = plan**: dry-run shows the migration list; user must pass `--apply`.
  - **Safe failure modes**: refuses to run on a non-harness directory (no AGENTS.md or no docs/).
- **Why**:
  - The most common upgrade path is "I bootstrapped 6 months ago with v1.0.0, now v1.4.0 has new things, what do I do?". Before this script, the answer was "read CHANGELOG.md and manually apply diffs". Now: run `sync-project.sh --apply`.
  - **Fence-block strategy is the only safe way to patch user-owned files**: open-coding the entire AGENTS.md would clobber user content; not patching at all leaves new capabilities invisible. Fences let the harness own its named sections without competing with the user.
  - **`--apply` is opt-in, not default**: a sync that accidentally clobbers a project is worse than no sync. Default to showing the plan.
- **Consequence**:
  - A user bootstrapped at v1.0.0 who upgrades to v1.4.0 can now run `bash scripts/sync-project.sh --apply` and get: `.harness-state.json` written, missing GitHub templates copied, AGENTS.md patched (only the fenced block), compact-report.json back-filled in each evidence dir.
  - **All sync actions are idempotent and tested**: 12 bats tests in `tests/sync-project.bats` cover help, refusal modes, dry-run, --apply, idempotency, AGENTS.md preservation, compact-report non-overwrite, status drift reporting.
- **Revisit when**:
  - The harness introduces a new piece of project state that requires user input during sync (e.g., "which agent should own this evidence dir?"). Today everything is automatic.
  - A user files a bug about fence-block corruption — the Python heredoc is the most fragile part; consider switching to a dedicated Python file.

## D-016 · Auto-update flow: 3 paths, all idempotent, all non-destructive

- **Status**: durable
- **Context**: After shipping v1.8.0, the user asked "I have projects I took over with the skill — do they auto-update?" The answer: no, the user has to run something. The first version (v1.8.0) assumed the bootstrap workflow always wrote `.harness-state.json`. That assumption was wrong: any project bootstrapped before v1.4.0 (when state-file was introduced) had everything EXCEPT the state file. v1.8.0 audit found 12 such projects in the user's `~/repos` — full harness-style structure (AGENTS.md + docs/evidence/ + .github/ISSUE_TEMPLATE/) but NO state file. They were "harness-managed but un-registered."
- **Decision**: Three user-facing paths cover all auto-update scenarios. All idempotent. All non-destructive:
  1. **`npx -y skills update lora-sys/ai-engineering-harness -g`** — updates the install location (5 CLI agent dirs). This is the "the skill itself gets updated" path. Fully automatic.
  2. **`bash scripts/sync-project.sh --project-dir <proj> --auto`** — updates a single project. Re-runs all migrations, but each is idempotent (state-file: timestamp-only diff; AGENTS.md fenced block: replaced, not appended; .github templates: copied only if missing; compact-report.json: never overwrites). For projects that have `.harness-state.json` already.
  3. **`bash scripts/register-existing.sh <root>`** — bulk-register all un-registered harness projects under a directory tree. For projects that pre-date v1.4.0's state file. Walks the tree, finds every project with AGENTS.md + docs/evidence/, runs `--auto` on the ones that don't have a state file. Idempotent.
- **Why three paths, not one**: they cover three different real situations:
  - "I just upgraded the harness" → #1, then #2 per project, OR #3 if you have many un-registered.
  - "I have an old project that never got registered" → #3 once.
  - "I have a new project I just bootstrapped" → state file is written by bootstrap, then #2 is a no-op (idempotent).
- **Critical design property**: the auto-update is **non-destructive**. Migrations never overwrite user content. The fenced-block pattern (`<!-- HARNESS:START name -->`) is the boundary between "harness owns this" and "user owns this" — sync only touches the harness-owned block. compact-report.json back-fill is gated on the source `implementation-report.md` existing, so we never invent data.
- **Consequence**:
  - User experience: upgrade the harness, then run `register-existing.sh ~/repos` (or sync per-project). One command per "tier" of fix.
  - No more "silent skip" of pre-v1.4.0 projects. The audit script `/tmp/audit-harness-projects.sh` makes the missing-state-file case visible.
- **Revisit when**:
  - A new state-tracking mechanism is introduced (next time, write the bulk-register script at the same time, not after-the-fact).
  - A new migration is added that DOES need to overwrite user data (e.g., a `.eslintrc` reformat). The non-destructive principle may need to be relaxed with explicit consent.
  - The audit script grows: as more "I have projects like X" questions come up, add a column to the audit table.

## D-017 · Cross-version regression test (the first eval layer)

- **Status**: durable
- **Context**: The harness has multiple "eval" layers:
  1. Project-level eval (per-Issue: QA agent, adversarial review, evidence pack, CI gate) — v0.0+
  2. Harness-self-test (80 bats tests for the harness's own scripts) — v1.3.0
  3. **Cross-version regression** (does upgrading break user projects?) — NEW in v1.9.0
  4. Agent regression (do agents behave the same across versions?) — NOT YET
  5. Awwwards auto-scoring (LLM-as-judge on rendered pages) — NOT YET
  6. Skill-level benchmark (golden projects, known-good outputs) — NOT YET
- **Decision**: Build the cross-version regression as the FIRST eval layer because:
  - It's the cheapest (no LLM calls, no benchmark data sets, no human raters)
  - It catches a class of bugs the other layers can't (migration scripts that quietly change behavior across versions)
  - It uses the harness's own `sync-project.sh` as the system-under-test, so any regression in the migration itself is caught
- **Test design** (`tests/cross-version/`):
  1. Fixture projects (`project-alpha` pre-v1.4, `project-beta` post-v1.6) with realistic user content + AGENTS.md + docs/evidence/
  2. Frozen v1.7.0 `sync-project.sh` extracted via `git show v1.7.0:scripts/sync-project.sh`
  3. Run v1.7 then HEAD on each fixture, assert invariants:
     - User content outside the harness fenced block is preserved (REGRESSION check)
     - HEAD's fenced block is non-empty (capabilities must be listed)
     - State file version == HEAD version
     - Both versions are idempotent (re-run produces same output)
     - Initial state v1.6 is correctly upgraded to HEAD
- **Why this design**:
  - The fence-block is the test for "user content preservation" — the test would have caught any regression in `set_fenced` (Python heredoc that strips/replaces the harness block without touching user content outside).
  - The idempotency assertion catches "second run breaks things" bugs (e.g., the v1.8.0 del-keyError bug we had early in this session).
  - The frozen v1.7 script + the HEAD script comparison is the actual cross-version check.
- **Consequence**:
  - Any future migration change MUST pass this test before merging.
  - The test catches the "we clobbered user content" class of bug — the #1 thing users would complain about.
  - It's a 1-second test. No LLM, no benchmark data, no human. Just diff.
- **Revisit when**:
  - Migration semantics change (e.g., a new state file field) — update expected outputs, but the invariant checks should still pass.
  - New fixtures are added (e.g., a v0.x pre-`sync-project.sh` project that came in via `npx skills add`).
  - The other eval layers (agent regression, Awwwards scoring) get built — they're orthogonal to this one.
